【 MySQL 系列 02 】：事务
发表于 2019-08-31| 更新于 2020-03-21|数据库
阅读量:

一、事务

1、事务的基础概念

事务就是一组原子性的 SQL 操作，或者说一个独立的工作单元。

如果数据库引擎能够成功地对数据库应用该组操作的全部语句，那么就执行该组操作。如果其中有任何一条语句因为崩溃或其他原因无法执行，那么所有的语句都不会执行。

也就是说，事务内的语句，要么全部执行成功，要么全部执行失败。

一个运行良好的事务处理系统，必须具备 ACID 特征。

ACID 表示 原子性（Atomicity）、一致性（Consistency）、持久性（Durability）、隔离性（Isolation）。

2、事务的分类

从事务理论的角度来说，可以把事务分为以下几种类型：

  • 扁平事务（ Flat Transactions ）
  • 带有保存点的扁平事务（ Flat Transactions with Savepoints ）
  • 链事务（ Chained Transactions ）
  • 嵌套事务（ Nested Transactions ）
  • 分布式事务（ Distributed Transactions ）

3、三个隐藏字段

innoDB 向数据库中存储的每行添加了三个隐藏字段。

DB_TRX_ID 事务id

  • 占 6 字节，表示这一行数据最后插入或修改的事务 id 。此外删除在内部也被当作一次更新，在行的特殊位置添加一个删除标记（ 记录头信息有一个字节存储是否删除的标记 ）。

DB_ROLL_PTR 回滚指针

DB_ROW_ID 行ID

  • 占 7 字节，它就像自增主键一样随着插入新数据自增。如果表中不存主键 或者 唯一索引，那么数据库就会采用 DB_ROW_ID 生成聚簇索引。否则 DB_ROW_ID 不会出现在索引中。

二、ACID

在事务的四个特征中，一致性是事物的根本要求，而某些情况会对事务的一致性造成破坏 ：

  • 事务的并发执行
  • 事务故障或系统故障

数据库系统通过并发控制技术和日志恢复技术来避免这种情况的发生

  • 并发控制技术保证了事务的隔离性，使数据库的一致性状态不会因为并发执行的操作而破坏。
  • 日志恢复技术保证了事务的原子性，使一致性状态不会因事务或系统故障被破坏。同时使已提交的修改不会因系统崩溃而丢失，保证了事务的持久性。

原子性

一个事务必须被视为一个不可分割的最小工作单元，整个事务中的所有操作要么全部提交成功，要么全部失败回滚，对于一个事务来说，不可能只执行其中的一部分操作，这就是事务的原子性。

一致性

事务的执行结果必须使数据库从一个一致性状态到另一个一致性状态。

持久性

事务一旦提交，其对数据库的更新就是持久的。任何事务或系统故障都不会导致数据丢失。

隔离性（ 四种隔离级别 ）

事务具有隔离性，理论上来说并发执行的事务不会相互影响，一个事务所做的修改在最终提交前，对其它事务是不可见的。

然而，完全的隔离性会导致系统并发性能很低，降低对资源的利用率，因而实际上会对数据库的隔离性适当放宽，导致数据库的一致性降低。

read uncommitted（ 读未提交 ）

  • 在 read uncommitted 级别。事务可以读取其它事务未提交的数据，这也被称为脏读（Dirty Read）。

read committed（ 读已提交 ）

  • 在 read committed 级别。事务可以读取到其它事务已提交的数据。

    该级别是大多数数据库系统的默认隔离级别（但 MySQL 不是）。

repeatable read（ 可重复读 ）

  • 该级别保证了同一事务中多次读取相同的记录结果是一致的。

    该级别是 MySQL 默认隔离级别。

serializable（ 串行化 ）

  • 该级别是最高的隔离级别。它通过强制事务串行执行，避免了幻读。简单来说，serializable 会在读取的每一行数据上都加锁，这可能导致大量的超时和锁争用的问题。

不同隔离级别可能导致的问题

三、事务的实现原理

原子性、持久性 ：

  • redo log 称为重做日志，用来保证事务的原子性和持久性。

一致性 ：

  • undo log 用来保证事务的一致性。（ 帮助事务回滚及 MVCC 的功能 ）

隔离性 ：

  • 通过锁 + MVCC 实现。

有的 DBA 或许会认为 undo 是 redo 的逆过程，其实不然。

  • redo 和 undo 的作用都可以视为是一种恢复操作，redo 恢复提交事务修改的页操作，而 undo 回滚行记录到某个特定版本。
  • 因此两者记录的内容不同：
    • redo 通常是物理日志，记录的是页的物理修改操作。
    • undo 是逻辑日志，根据每行记录进行记录。

redo log

redo log 记录的是新数据的备份。在事务提交前，只要将 redo log 持久化即可，不需要将数据持久化。当系统崩溃时，虽然数据没有持久化，但是 redo log 已经持久化。系统可以根据 redo log 的内容，将所有数据恢复到最新的状态。

fsync()

undo log

undo log 会产生 redo log，也就是 undo log 的产生会伴随着 redo log 的产生，这是因为 undo log 也需要持久性的保护。

undo log 又可以细分为：

  • insert undo log

    记录了 insert 操作 , 只在事务回滚时需要 , 并在事务提交后便可丢弃 。

  • update undo log

    记录了 update 、delete 操作 , 在 事务回滚 和 快照读 时使用 , 当数据库所使用的快照中不涉及该日志记录 , 才会被删除 。

undo log 实现事务回滚的原理

Undo log 是逻辑日志，它记录的内容可以理解为：

  • 当 delete 一条记录时，Undo log 中会记录一条对应的 insert 记录。

  • 当 update 一条记录时，Undo log 中会记录一条对应相反的 update 记录。

  • 等，类似相反的操作日志。

undo log 实现 MVCC 的原理

  • 当用户读取一行记录时，若该记录已经被其他事务占用，当前事务可以通过 undo log 读取之前的行版本信息，以此实现非锁定读取。

purge

对于上述的 delete 操作，通过前面关于 undo log 的介绍已经知道仅是将主键列等于 1 的记录 delete flag 设置为 1，记录并没有被删除，即记录还是存在于 B+ 树中。其次，对辅助索引上 a 等于 1，b 等于 1 的记录同样没有做任何处理，甚至没有产生 undo log 。而真正删除这行记录的操作其实被“延时”了，最终在 purge 操作中完成。

purge 用于最终完成 delete 和 update 操作。这样设计是因为 InnoDB 存储引擎支持 MVCC ，所以记录不能在事务提交时立即进行处理。这时其他事物可能正在引用这行，故 InnoDB 存储引擎需要保存记录之前的版本。而是否可以删除该条记录通过 purge 来进行判断。若该行记录已不被任何其他事务引用，那么就可以进行真正的 delete 操作。可见，purge 操作是清理之前的 delete 和 update 操作，将上述操作“最终”完成。而实际执行的操作为 delete 操作，清理之前行记录的版本。

锁

1、锁的分类

在 MySQL 中，锁可以分为两类：

  • 共享锁：共享锁定是将对象数据变为只读形式，不能进行更新。也称为：读锁

  • 排他锁：排他锁定是执行 insert / update / delete 时，其他事务不能读取该数据。也称为：写锁

2、锁的粒度

  • 记录

  • 表

  • 数据库

3、基于锁的并发控制

  • 流程：

    • 1、事务根据自己对数据项进行的操作类型申请相应的锁（读申请共享锁，写申请排他锁）

    • 2、申请锁的请求被送到锁管理器。锁管理器根据当前数据项是否已经有锁以及申请的和持有的锁是否冲突决定是否为该申请授予锁。

    • 3、若锁被授予，则申请锁的事务可以继续执行，反之，事务将进行等待，直到获取锁。

  • 可能产生的问题：

    死锁

  • 除了锁，还有其它技术能实现并发控制

    • 基于时间戳

    • 基于有效性检查

    • 基于快照隔离

  • 表象 : 快照读 ( 非阻塞读 ) — 伪 MVCC
  • 内在 : next-key 锁 ( 行锁 + gap 锁 )

  1. 数据行里的 DB_TRX_ID 、DB_ROLL_PTR 、DB_ROW_ID 字段

Gap 锁

  • 对主键索引或唯一索引会用 Gap 锁吗 ?
