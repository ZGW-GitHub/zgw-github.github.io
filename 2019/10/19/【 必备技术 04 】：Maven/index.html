一、什么是 Maven

Maven 是一个项目管理工具

有两个核心功能 :

  • 依赖管理

  • 项目构建

    Maven 自身集成了 tomcat 插件 , 可以对项目进行 编译 、测试 、打包 、安装 、发布 等操作

二、Maven 仓库

  • 本地仓库
  • 远程仓库 ( 私服 )
  • 中央仓库

三、常用命令

mvn compile

  • 编译 maven 项目
  • 将 src/main/java 目录下的文件编译为 class 文件输出到 target 目录下

mvn test

  • 测试 maven 项目
  • 执行 src/test/java下的单元测试类
  • 并隐式执行 mvn compile

mvn package

  • 打包 maven 项目
  • 对 java 工程打成 jar 包 , 对 web 工程打成 war 包
  • 并隐式执行 compile 、test

mvn install

  • 安装 maven 项目
  • 将打包后的项目 ( jar / war 包 ) 发布到本地仓库
  • 并隐式执行 compile 、test 、package

mvn deploy

  • 发布 maven 项目
  • 并隐式执行 compile 、test 、package 、deploy

mvn clean

  • 删除 target 目录

四、Maven 生命周期

  1. 默认生命周期
    • mvn compile
    • mvn test
    • mvn package
    • mvn install
    • mvn deploy
  2. 清理生命周期
    • mvn clean
  3. 站点生命周期

五、坐标

  • groupId + artifactId + version 构成了依赖的基本坐标
  • scope 定义了依赖的范围，默认是 compile
  • optional 定义了是否为可选依赖，默认是 false
  • exclusion 用来排除传递性依赖

六、Maven 依赖

直接依赖

在 pom 文件中 , 因直接引入坐标而导入的 jar 包

间接依赖

在 pom 文件中 , 因其坐标的依赖而导入的 jar 包

七、依赖冲突

1、何为依赖冲突

当我们的直接依赖与传递依赖导入了相同的 jar 包 , 且 jar 包版本不同的情况下就会发生依赖冲突

2、依赖冲突的解决

a、Maven 默认处理策略

最短路径优先

Maven 面对 D1 和 D2 时，会默认选择最短路径的那个 jar 包，即 D2 。

E -> F -> D2 比 A -> B -> C -> D1 路径短 1

最先声明优先

如果路径一样的话

如 : A -> B -> C1 , E -> F -> C2 ，两个依赖路径长度都是 2，那么就选择最先声明

八、依赖范围

在正式介绍依赖范围之前，我们需要知道一件事，那就是：

Maven 项目的环境变量(classpath)有三种，分别为 : 编译时用的环境变量、测试时用的环境变量和运行时用的环境变量。

而依赖范围就是用来控制依赖与这三种环境的关系的 ：

  • compile：编译依赖范围，此为默认值，对编译、测试和运行三种环境变量都有效 。
  • test：测试依赖范围，仅对测试环境变量有效 。
  • provided：已提供的依赖范围，对编译和测试环境变量都有效 。
  • runtime：运行依赖范围，对测试和运行环境变量都有效 。
  • system：系统依赖范围，对编译和测试环境变量都有效，但由于此依赖不是通过 Maven 仓库解析的，而且往往与本机系统绑定，可能造成构建的不可移植，因此应该谨慎使用 。
  • import：导入依赖范围，并不会对上面的三种环境变量产生实际的影响 。

依赖范围 对 传递依赖的影响 :

观察下图，我们会发现：

  • 当第二直接依赖为 compile 时，则依赖关系以第一直接依赖为准；
  • 当第二直接依赖为 test 时，则没有依赖关系；
  • 当第二直接依赖为 provided 时，只有第一直接依赖也为 provided 时才有 provided 范围的依赖关系；
  • 当第二直接依赖为 runtime 时，除了第一直接依赖关系为 compile 时依赖范围为 runtime，其他三种皆与第一直接依赖范围相同 。
