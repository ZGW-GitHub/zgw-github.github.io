【 Java 系列 01 】：Java web 核心技术

一、Web 理论

1、软件架构

  • C / S 架构 : 客户端 / 服务器
  • B / S 架构 : 浏览器 / 服务器

2、资源分类

静态资源

如 : html 、css 、JavaScript 、png …

即 :

  • 所有用户访问得到的结果都是一样的
  • 静态资源可以直接被浏览器解析

动态资源

如 : servlet / jsp 、php 、asp …

即 :

  • 每个用户访问相同的资源得到的结果可能不同
  • 动态资源在被访问时 , 需要先转化为静态资源 , 再返回给浏览器

3、网络通信三要素

  • IP
  • 端口
  • 传输协议

二、Web 服务器

1、什么是服务器

安装了服务器软件的计算机

2、什么是服务器软件

接收并处理用户请求 , 并作出响应

3、什么是Web服务器软件(Web容器)

  • 接收并处理用户请求 , 并作出响应
  • 在 web 服务器软件中 , 可以部署 web 项目 , 让用户通过浏览器来访问这些项目

4、常用的 web 服务器软件

Tomcat

  • Apache 基金组织
  • 中小型的 JavaEE 服务器 , Servlet 容器
  • 支持少量的 JavaEE 规范 , 支持 Servlet / jsp 规范
  • 开源 , 免费

Nginx

  • 是异步框架的 web 服务器 , 也可以用作 反向代理 , 负载平衡器 和 HTTP缓存

webLogic

  • oracle 公司
  • 大型的 JavaEE 服务器
  • 支持所有的 JavaEE 规范
  • 收费

webSphere

  • IBM 公司
  • 大型的 JavaEE 服务器
  • 支持所有的 JavaEE 规范
  • 收费

JBOSS

  • JBOSS 公司
  • 大型的 JavaEE 服务器
  • 支持所有的 JavaEE 规范
  • 收费

三、Tomecat

1、部署项目

a、方式一

直接将 .war 格式的项目文件放到 webapps 目录下即可

该方式部署 : 访问路径(虚拟路径) 与项目路径一致

步骤 : 将项目文件压缩 , 修改格式后缀为 war , 将文件移动到 webapps 目录下 . 此时 war 包会自动解压缩 .

b、方式二

修改 conf 目录下 的 server.xml 文件 , 在 标签体中配置 <Context docBase="项目存放的路径" path="虚拟路径" />

该方式部署 : 访问路径(虚拟路径) 与项目路径可以不一致

c、方式三

在 conf/Catalina/localhost 目录下创建任意名称的 xml 文件 . 并在文件中编写 <Context docBase="项目存放路径" />

该方式部署 : 访问路径(虚拟路径) 为所创建的 xml 文件的文件名

四、Servlet (Server applet) - JavaWeb 三大组件之一

JavaWeb 三大组件之一

Servlet 必须依赖于服务器才能运行 , 它是由服务器来执行的

实现了 Servlet 接口的类就是 Servlet

1、Servlet 类继承体系

  • Servlet 的实现类 : GenericServlet 抽象类
  • GenericServlet 的实现类 : HttpServlet 抽象类

  • GenericServlet 抽象类 :

    • 作用 :

      将除 service() 方法的其余方法做了实现 , 只保留 service() 方法仍为抽象方法

  • HttpServlet 抽象类 :

    • 作用 :

      将 Servlet 判断如何处理 Http 请求响应(即 : get / post …)的操作简化了 , 使得 Servlet 不再疲于这些冗余操作了

    • 使用 :

      定义类继承其 , 并重写其方法

2、Servlet 的生命周期

init 方法

Servlet 被创建时执行 , 只执行一次 , 一般用于资源申请

Servlet 什么时候被创建 ?

  • 默认情况下 : 第一次被访问时创建
  • 可修改为 : 在服务启动时创建

service 方法

即 : void service ( ServletRequest req , ServletResponse res ) 方法 , 执行多次 , 用于提供服务

每次访问 Servlet 时 , Servcie 方法都会被调用一次

destroy 方法

Servlet 被销毁时执行 , 只执行一次 , 一般用于资源释放

如 : 服务器正常关闭时 , 会执行一次 destroy 方法

3、Servlet 是单例的

因为 Servlet 的 init 方法只执行一次 , 说明一个 Servlet 在内存中只存在一个对象 , Servlet 是单例的

因此多个用户同时访问时 , 可能存在线程安全问题

解决 :

  • 尽量别在 Servlet 中定义成员变量 , 因为多线程(用户)操作中 , 对象的成员变量是线程共享的
  • 即使定义了成员变量也不要对其值进行修改 , 此时即便多线程(用户)访问 , 也都是读 , 没有了写也就没有了线程安全问题

4、Servlet 3.0

优点 :

支持了注解配置 , 可以舍弃 web.xml 了

响应行

格式 :

协议/版本 响应状态码 状态码描述

响应状态码
分类 分类描述
1** 信息，服务器收到请求，需要请求者继续执行操作
2** 成功，操作被成功接收并处理
3** 重定向，需要进一步的操作以完成请求
4** 客户端错误，请求包含语法错误或无法完成请求
5** 服务器错误，服务器在处理请求的过程中发生了错误
状态码 状态码英文名称 中文描述
100 Continue 继续。客户端应继续其请求
101 Switching Protocols 切换协议。服务器根据客户端的请求切换协议。只能切换到更高级的协议，例如，切换到HTTP的新版本协议
200 OK 请求成功。一般用于GET与POST请求
201 Created 已创建。成功请求并创建了新的资源
202 Accepted 已接受。已经接受请求，但未处理完成
203 Non-Authoritative Information 非授权信息。请求成功。但返回的meta信息不在原始的服务器，而是一个副本
204 No Content 无内容。服务器成功处理，但未返回内容。在未更新网页的情况下，可确保浏览器继续显示当前文档
205 Reset Content 重置内容。服务器处理成功，用户终端（例如：浏览器）应重置文档视图。可通过此返回码清除浏览器的表单域
206 Partial Content 部分内容。服务器成功处理了部分GET请求
300 Multiple Choices 多种选择。请求的资源可包括多个位置，相应可返回一个资源特征与地址的列表用于用户终端（例如：浏览器）选择
301 Moved Permanently 永久移动。请求的资源已被永久的移动到新URI，返回信息会包括新的URI，浏览器会自动定向到新URI。今后任何新的请求都应使用新的URI代替
302 Found 临时移动。与301类似。但资源只是临时被移动。客户端应继续使用原有URI
303 See Other 查看其它地址。与301类似。使用GET和POST请求查看
304 Not Modified 未修改。所请求的资源未修改，服务器返回此状态码时，不会返回任何资源。客户端通常会缓存访问过的资源，通过提供一个头信息指出客户端希望只返回在指定日期之后修改的资源
305 Use Proxy 使用代理。所请求的资源必须通过代理访问
306 Unused 已经被废弃的HTTP状态码
307 Temporary Redirect 临时重定向。与302类似。使用GET请求重定向
400 Bad Request 客户端请求的语法错误，服务器无法理解
401 Unauthorized 请求要求用户的身份认证
402 Payment Required 保留，将来使用
403 Forbidden 服务器理解请求客户端的请求，但是拒绝执行此请求
404 Not Found 服务器无法根据客户端的请求找到资源（网页）。通过此代码，网站设计人员可设置”您所请求的资源无法找到”的个性页面
405 Method Not Allowed 客户端请求中的方法被禁止
406 Not Acceptable 服务器无法根据客户端请求的内容特性完成请求
407 Proxy Authentication Required 请求要求代理的身份认证，与401类似，但请求者应当使用代理进行授权
408 Request Time-out 服务器等待客户端发送的请求时间过长，超时
409 Conflict 服务器完成客户端的 PUT 请求时可能返回此代码，服务器处理请求时发生了冲突
410 Gone 客户端请求的资源已经不存在。410不同于404，如果资源以前有现在被永久删除了可使用410代码，网站设计人员可通过301代码指定资源的新位置
411 Length Required 服务器无法处理客户端发送的不带Content-Length的请求信息
412 Precondition Failed 客户端请求信息的先决条件错误
413 Request Entity Too Large 由于请求的实体过大，服务器无法处理，因此拒绝请求。为防止客户端的连续请求，服务器可能会关闭连接。如果只是服务器暂时无法处理，则会包含一个Retry-After的响应信息
414 Request-URI Too Large 请求的URI过长（URI通常为网址），服务器无法处理
415 Unsupported Media Type 服务器无法处理请求附带的媒体格式
416 Requested range not satisfiable 客户端请求的范围无效
417 Expectation Failed 服务器无法满足Expect的请求头信息
500 Internal Server Error 服务器内部错误，无法完成请求
501 Not Implemented 服务器不支持请求的功能，无法完成请求
502 Bad Gateway 作为网关或者代理工作的服务器尝试执行请求时，从远程服务器接收到了一个无效的响应
503 Service Unavailable 由于超载或系统维护，服务器暂时的无法处理客户端的请求。延时的长度可包含在服务器的Retry-After头信息中
504 Gateway Time-out 充当网关或代理的服务器，未及时从远端服务器获取请求
505 HTTP Version not supported 服务器不支持请求的HTTP协议的版本，无法完成处理

大小限制 大小有限制 ( 通常 4kb 内 )

Cookie

1、Cookie 原理

基于响应头 set-cookie 和请求头 cookie 实现

  1. 服务端 : 自动将 Cookie 中的数据 , 封装到响应头的 set-cookie 中 , 发送给客户端 , 如 : set-cookie:msg=hello

  2. 客户端 : 收到后将 set-cookie 中的数据保存下来

  3. 客户端 : 客户端再次发送请求 , 自动将这些数据封装到请求头 cookie 中 , 发送给服务器 , 如 : cookie:msg=hello

2、Cookie 的细节

a、一次能否发送多个 Cookie : 可以

客户端 :

  • 请求时发送的多个 Cookie 的数据会存储在一个请求头中 , 即 : cookie

服务端 :

  • 可以通过创建多个 Cookie 对象 , 使用 response 调用多次 addCookie 方法发送 Cookie 即可 , 而响应时它们的数据是分开存储的 , 即 : 有多个响应头 , 而且键都为 set-cookie , 键的取值为各个 Cookie 的数据
b、Cookie 在浏览器中保存多长时间

默认情况下 :

  • 当浏览器关闭后 , Cookie 数据即销毁

可以设置持久化存储 Cookie :

  • 在服务端使用 Cookie 对象调用 setMaxAge(int seconds) 方法

    参数为正数 : 持久化并指定存活时间

    参数为负数 : 默认

    参数为 0 : 删除 cookie 数据

c、Cookie 能不能存中文

  • tomcat 8 之前 , Cookie 中不能直接存储中文数据 , 要将中文数据进行转码 ( 一般采用 URL 编码 )

  • tomcat 8 之后 , Cookie 支持中文数据

d、Cookie 共享问题

  • 同一个 tomcat 服务器部署了多个 web 项目 , 它们的 Cookie 能否共享 ?

    • 默认不能

    • 通过设置 Cookie 的获取范围 , 可以实现共享

      cookie.setPath(String path)

  • 不同 tomcat 服务器部署了多个 web 项目 , 它们的 Cookie 能否共享 ?

    • 默认不能

    • 通过设置 Cookie 的获取范围 , 可以实现共享

      cookie.setDomain(String url)

3、Cookie 的使用

  • 创建：

    • new Cookie(String name , String value)

  • 发送：

    • response.addCookie(Cookie cookie)

  • Cookie 中数据的获取

    • 获取请求中的所有 Cookie

      一个键值对就是一个cookie

      • Cookie[] cookies = request.getCookies()

    • getName() 方法，获取 Cookie 的名字（键）

      • String string = Cookie.getName()

    • getValue() 方法，获取 Cookie 的值

      • String string = Cookie.getValue()

Session

1、Session 原理

Session 的实现是依赖于 Cookie 的

  1. 服务端 : 当请求头 cookie:XXX:XXX,YY:YY 中没有 session 的键值对信息( JSESSIONID=xxxxx ) 时 , 说明没有 Session 对象 , 则当服务器获取 Session 时服务器会新建一个 Session . 并将 Session 的唯一标识( JSESSIONID=xxxxx ) 保存在响应头的 set-cookie 中发送给客户端
  2. 客户端 : 再次发送请求时 , 会将 Session 的唯一标识保存在请求头的 cookie 中发送给服务器
  3. 服务端 : 服务端收到请求 , 从请求头中获取到 Session 的唯一标识 SessionId

2、Session 的细节

a、服务器不关闭 , 客户端重启前后获取的两次 Session 是同一个吗 ?

默认 :

不是 , 因为 Cookie 默认是在浏览器关闭后即销毁 , 所以客户端重启后 , 获取的两次 Session 都是服务端新创建的

但是 :

可以通过设置 Cookie : JSESSIONID=xxxxx 的存活时间持久化来实现客户端重启获取同一个 Session

b、客户端不关闭 , 服务器重启前后获取的两次 Session 是同一个吗 ?

不是同一个

但是为了用户体验 , 要通过以下手段确保数据不会丢失 ( Tomcat 会自动完成以下操作 ) :

  • session 的钝化 :

    在服务器正常关闭前 , 将 Session 对象序列化到硬盘上

  • session 的活化 :

    在服务器启动后 , 将 Session 文件序列化为 Session 对象

3、Session 的使用

  • 获取 HttpSession 对象
    • HttpSession session = request.getSession()
  • 使用 HttpSession 对象
    • Object getAttribute(String name)
    • void setAttribute(String name , Object value)
    • void removeAttribute(String name)

八、ServletContext - Context 域

  • ServletContext 对象用来存放全局变量 , 每个 Java 虚拟机的每个 Web 项目只有一个 ServletContext 对象

  • ServletContext 对象由 Web 服务器创建 , 以保证它的唯一性

使用 :

获取：

  • request.getServletContext()
    通过 request 对象获取
  • this.getServletContext()
    通过 HttpServlet 获取

数据操作

  • void setAttribute(String name , Object value)
    存储数据
  • Object getAttitude(String name)
    通过键获取值
  • void removeAttribute(String name)
    通过键移除键值对

doFilter

会执行多次

destory

只执行一次 , 常用于释放资源
