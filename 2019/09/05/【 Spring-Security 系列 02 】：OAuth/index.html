一、OAuth 协议

1、OAuth 协议解决的问题

实现不同厂商的应用之间用户的授权与资源共享的安全

2、OAuth 协议授权流程

  • 服务提供商（Provider）：提供令牌的一方，比如：QQ 、微信
  • 资源所有者（Resource Owner）：即用户，我们要获取的即用户的资源。
  • 第三方应用（Client）：指获取授权的应用，一般就是我们自己开发的应用。
  • 认证服务器（Authorization Server）：即服务提供商专门用来处理认证的服务器，认证用户的身份并产生令牌。
  • 资源服务器（Resource Server）：即服务提供商存放用户的资源的服务器。认证服务器和资源服务器虽然是两个角色，但他们一般也可以在同一个应用，同一台机器上。

3、OAuth 协议的授权模式

①、授权码模式

a、授权码模式与其它模式的区别

用户需要先申请授权码，再携带授权码去申请令牌；

而且，用户的同意授权是在服务提供商处理的。

b、授权码模式流程图

c、授权码模式步骤 一 ：获取授权码
  • 获取授权码 GET
    • *URI *：/oauth/authorize
    • 请求参数 ：
      • response_type ：code
      • client_id ：
      • redirect_uri ：授权码的重定向地址，如：（ 可选 ）
        • 填写：http://snow.com
        • 重定向地址：http://snow.com/?code=XXX
      • scope ：（ 可选 ）
      • state ：（ 推荐 ）
d、授权码模式步骤 二 ：获取令牌
  • 获取令牌 POST
    • URI ：/oauth/token
    • 请求头 ：
      • Authorization ：Basic XXX ( Base64 加密后的 client_id 与 client_secret )
      • Content-Type ：application/x-www-form-urlencoded
    • 请求参数 ：
      • grant_type ：authorization_code
      • code ：
      • redirect_uri ：重定向地址，要写和获取授权码时参数 redirect_uri 的取值一样
      • client_id ：
      • scope ：
    • 返回值 ：
      • access_token
      • token_type
      • refresh_token
      • expires_in
      • scope
e、授权码模式步骤 三 ：获取用户信息
  • 获取用户信息 GET
    • 请求头 ：
      • Authorization ：token_type token （ 填写具体的 token_type 、token ）

②、简化模式

简化模式授权码模式相比，简化模式返回的授权码是返回到网页的，授权码模式是将令牌返回至一个服务器的。而且没有授权码步骤。

三、Spring Social 与 Spring Security OAuth

  • Spring Social

    Spring Social 封装了第三方客户端访问服务提供商时要做的大部分操作。

  • Spring Security OAuth

    Spring Security OAuth 封装了服务提供商为第三方客户端提供令牌所要做的绝大部分操作，使用 Spring Security OAuth 我们可以快速搭建起一个服务提供商程序。

三、Spring Social

1、Spring Social 基本原理

Spring Social 是基于 OAuth 协议来提供功能的。

它将 OAuth 的认证流程和获取用户信息的步骤都封装到了一个过滤器中。并将该过滤器加入 Spring Security 的过滤器链中。用于给应用提供社交登录与认证的服务。

a、SocialAuthenticationFilter

通过 requiresAuthentication 方法可知：该过滤器处理的请求 URL 必须为：{filterProcessesUrl}/{providerId} 形式的。

通过认证方法 attemptAuthentication 可以得知，它首先会调用 detectRejection 方法，检测用户是否拒绝授权，若用户拒绝则抛异常；否则，会获取 providerId 及对应的认证服务类，来处理认证，认证失败，则抛异常。

通过查看执行认证的方法 attemptAuthService 的注释可以看出，第一次请求，会抛异常来重定向到服务提供商提供的认证地址去。在用户确认授权后，重定向回来时，就是第二次请求了，就回带着授权码去服务提供商获取令牌。

b、OAuth2AuthenticationService

在获取 SocialAuthenticationToken 的方法中可以看出，当请求中无 code(授权码)，就重定向到服务提供商。通过：buildReturnToUrl 和 buildAuthenticateUrl 可以看出，它会自动帮我们构造回调地址以及重定向到认证服务器的地址。所以，本地测试要使用域名访问，可以在 hosts 中配置域名映射。否则你访问 localhost 是重定向不回来的，而且域名必须与QQ互联上配置的域名保持一致。

c、OAuth2Template

第二次请求时，有了授权码，则会用授权码去获取令牌 AccessGrant 用于构造 Connection，最终构造 SocialAuthenticationToken（注意此时的 SocialAuthenticationToken 是未认证的） 。

通过 exchangeForAccess 方法，可以发现，会自动帮我们带上获取令牌的参数，如果要带上 client_id、client_secret 需配置 useParametersForClientAuthentication=true。

获取到令牌后会自动帮我们将令牌封装到 AccessGrant 里，默认返回的数据结构为 Map，所以如果服务提供商返回令牌信息时不是 Map 结构的还需定制化处理。

d、OAuth2ConnectionFactory 与 AbstractConnection

创建好 AccessGrant 后，通过 OAuth2ConnectionFactory 创建 Connection，实际是创建 OAuth2Connection 对象。initApi() 方法会获取 ServiceProvider 中配置的 Api。

initKey() 用于生成服务提供商用户唯一的 key，根据 providerId 和 providerUserId(服务提供商的用户ID，即openId) 创建。而 providerUserId 则是通过 ApiAdapter 适配器来获取，这需要我们自行设置。

e、服务提供商认证完毕

获取到 SocialAuthenticationToken 后，相当于服务提供商那边认证完成，接着就会调用 doAuthentication 进行客户端用户认证。

与标准登录流程类似，同样可以自定义 AuthenticationDetailsSource；接着调用认证器进行认证，spring social 的认证器默认使用 SocialAuthenticationProvider 。

f、SocialAuthenticationProvider

在 authenticate 方法中，可以看出，它将会通过 Connection 封装的用户信息来获取 UserId(客户端UserId)，其中在这里 spring social 默认有一张表来存储 userId、providerId、providerUserId 之间的关系，可配置 JdbcUsersConnectionRepository 来维护对应的关系。

如果没有获取到对应的 userId，将抛出 BadCredentialsException，在 doAuthentication 里拦截到这个异常后，默认将重定向到 signupUrl 这个注册页面的地址，让用户先注册或绑定三方账号。signupUrl 默认为 "/signup"。

获取到对应的 userId 后，就根据 userId 查询用户信息，这需要我们自定义 SocialUserDetailsService 及 SocialUserDetails。获取到用户后，就会创建已认证的 SocialAuthenticationToken。

g、JdbcUsersConnectionRepository

通过 toUserId() 可以发现，根据 Connection 查找系统 userId 时，JdbcUsersConnectionRepository 默认的处理方式是：如果未查询到关联的 userId，可以自定义一个 ConnectionSignUp 用于注册用户并返回一个 userId，并且会调用 addConnection 添加关联。所以对于用户如果未注册，使用三方账号扫码自动注册用户的需求，就可以使用这种方式实现。

客户端这边认证成功后，就会通过 updateConnections 或 addConnection 将用户的 access_token、refresh_token、secret、用户和服务商的关联、等更新到数据库。

四、Spring Security OAuth

1、Spring Security OAuth 简介

①、为什么使用 Spring Security OAuth

前面实现的登录认证方式，登录成功后，登录信息是存储在服务器的 Session 里的，每次浏览器访问服务时，实际是在 Cookie 中带着 JSESSIONID 去访问服务，服务根据 JSESSIONID 来获取用户 Session，这种方式是基于服务器 Session 来保存用户信息。但在前后端分离或开发移动APP的时候，前端是单独部署在一台服务器上，用户实际访问的是 WebServer，所有的服务API请求再间接由 Web Server 发起。用户不再通过浏览器直接访问我们的后端应用，而是通过第三方的应用来访问。这种时候就不便于使用 Cookie + Session 的方式来保存用户信息，Cookie 存在跨域的问题，使用这种开发方式繁琐，安全性差。

于是就有了 Spring Security OAuth，类似于 QQ、微信认证那样，我们自己也可以作为服务提供商，前端应用或APP则作为第三方客户端，通过给客户端发放令牌，客户端在 http 参数中带着令牌来访问服务，服务端则通过令牌得到用户信息。Spring Social 封装了第三方客户端访问服务提供商时要做的大部分操作，而 Spring Security OAuth 则封装了服务提供商为第三方客户端提供令牌所要做的绝大部分操作，使用 Spring Security OAuth 我们可以快速搭建起一个服务提供商程序。

②、Spring Security OAuth 原理

要实现服务提供商程序，实际就是实现 认证服务器和资源服务器，作为认证服务器，可以使用 OAuth 的四种授权模式，来生成令牌并存储、发放。作为资源服务器，OAuth2 通过向 SpringSecurity 过滤器链上加入 OAuth2AuthenticationProcessingFilter 来对资源进行认证，解析令牌，根据令牌获取用户信息等。

  • OAuth2AuthenticationProcessingFilter 的作用：

从请求中取出 Token，根据存储策略从存储中找出用户信息，根据用户信息数据判断用户认证是否成功。

3、认证流程中用到的类源码详解

①、TokenEndpoint ：

可以理解为一个 Controller ，它会处理 /oauth/token 请求，是整个认证流程的入口点。

/oauth/token 请求的参数：

②、ClientDetailsService ：

可以与 UserDetailsService 相比，UserDetailsService 是读取用户信息的，ClientDetailsService 是读取第三方应用的信息的。

③、ClientDetails ：

与 UserDetails 对比，一个存用户信息，一个存第三方应用的信息。

④、TokenRequest ：

封装了请求中其它的相关参数，如：grant_type 、scope 、用户名密码登录时请求中的 username 、password ，也会把 ClientDetails 存入 TokenRequest 中。

⑤、TokenGranter ：

TokenRequest 会调用该接口，该接口封装了默认的四种授权模式和刷新模式，会根据请求中的 grant_type 挑一个具体的实现来执行令牌生成的逻辑。

其中：

  • CompositeTokenGranter ：类似于 AuthenticationManager

  • 其它五个模式 ( XXXGranter ) ：相当于 AuthenticationProperty

  • 注意：

    • 自定义的授权模式其 GrantType 为 ：bearer

⑥、OAuth2Request ：

在执行令牌的生成过程中会产生 OAuth2Request，它是 ClientDetails 与 TokenRequest 存储的信息的整合。

⑦、Authentication ：

在执行令牌的生成过程中会产生 Authentication ，存储了当前授权用户的一些信息，该信息是通过 UserDetailsService 来获取的

⑧、OAuth2Authentication ：

是 OAuth2Request 和 Authentication 组合成的，包含了当前是哪个第三方应用在请求哪个用户的授权，授权模式，以及授权的参数。

⑨、AuthorizationServerTokenServices ：

根据 OAuth2Authentication 中的信息生成一个 OAuth2AccessToken 。

⑩、TokenStore ：

令牌的存取

⑪、TokenEnhancer ：

令牌的增强器，可以用于改造令牌。
